What This Is

A MERN platform for managing a college fest from end to end. Events are created, published, and tracked through a full lifecycle. Organizers scan QR codes directly from any browser camera for live attendance. Teams form through cryptographic invite codes and coordinate via persistent real time chat. Merchandise approvals, anonymous feedback, and calendar export to three platforms are handled natively.

College fest management runs on spreadsheets, WhatsApp groups, and manual check in lines. This system replaces all of that.

Data Model, Discriminator Based Role Inheritance

The 9 MongoDB models center on a single design decision in the User schema. A single User base schema uses Mongoose discriminators. It extends into Participant, Organizer, and Admin subtypes instead of three separate collections.

Discriminators store all users in one MongoDB collection with a type tag. Fields shared across roles live on the base schema. Role specific fields live on the discriminator schemas. Population across discriminator types works without joining collections.

The alternative of separate collections would make cross role queries significantly messier. It would duplicate shared field definitions across three schemas.

Authentication and Access Control

JWT authentication with 7 day expiry. Tokens are stored in local storage and attached to every outgoing request via an Axios request interceptor. A response interceptor catches 401 errors and redirects to login globally.

The token payload carries user ID and role. On page load, jwt decode extracts the role from the stored token. This renders role appropriate UI instantly without an API call. Role based protected routes on both frontend and backend enforce access boundaries at both layers.

Real Time Systems, Two Socket.IO Implementations

Event Discussion Forum

Each event gets an isolated Socket.IO room. Messages from one event cannot leak into another. Access is registration gated. Only registered participants can post. Organizers can pin and delete any message. Participants can delete only their own messages. Emoji reactions use a MongoDB Map. Threaded replies store a reference to the parent message ID.

Team Chat

Teams that have finalized member selection get access to a persistent chat room. Messages are stored in MongoDB with a compound index for efficient retrieval. Members who join late can read previous discussions. An in memory Map tracks currently connected members per room.

Hackathon Team Registration

The team formation flow solves a specific UX problem. It allows people to join a team without sharing database ObjectIds.

The solution uses cryptographic 8 character invite codes. These are URL safe and not guessable from creation time. Member acceptance uses a 3 state model with explicit team leader control.

Finalization triggers a single batch operation. It iterates through all accepted members to create registration and ticket documents. It generates QR codes and sends confirmation emails. The operation is all or nothing.

Merchandise Approval Workflow

Automated payment verification is not feasible for college fests running on UPI transfers. The approval workflow handles this reality. Participants purchase an item and upload a payment proof image. The organizer reviews the image and approves or rejects.

Stock management uses atomic MongoDB increment operations to decrement available units. Atomicity prevents overselling when two organizers approve purchases simultaneously. The atomic decrement ensures stock never goes below zero.

Only approved purchases receive tickets. Only ticketed purchases are counted in attendance. The data pipeline is consistent from end to end.

Privacy By Design Feedback System

Participant identity is never stored in feedback documents. A SHA 256 hash of the participant and event ID serves as the deduplication key. This hash is mathematically infeasible to reverse. Organizers see ratings and comments but cannot identify the submitter.

A compound unique index prevents duplicate submissions without exposing identity. Feedback is only available for completed events and to participants who actually attended.

QR Attendance, Browser Native Scanning

Attendance scanning runs entirely in the browser. No native app installation is required. Organizers open their event dashboard on any device with a camera and scan continuously. The library handles permission requests and real time decoding.

Each attendance mark records a check in timestamp to enable time based analytics. Live statistics update in real time on the organizer dashboard.

Calendar Export, Three Platforms

Calendar export supports iCal file download, Google Calendar URL, and Microsoft Outlook URL. The iCal file generates standard output compatible with Apple Calendar and any desktop client.

The Google and Outlook URLs are constructed programmatically with URL encoded event parameters. All three options render as a dropdown accessible only to participants with confirmed tickets.

Library Selection Rationale

Every dependency in the backend has a documented justification. We use bcryptjs over native bcrypt for pure JavaScript portability across deployment environments. We use jsonwebtoken over session based authentication for stateless operation across multiple backend instances.

These decisions have clear rationale. Every library adds a dependency vector. The justification for each one is written down.
